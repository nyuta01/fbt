## Why fbt

`fbt` is a file build tool for operational knowledge work. It turns logs,
tickets, Markdown, spreadsheets, PDFs, prompts, rubrics, and generated manuals
into an inspectable local project with a manifest, build plan, immutable
artifact versions, review gates, and lineage.

The core stays small. It does not contain OCR, document conversion, LLM
providers, or agent runtimes. Those live behind external runners.

## From zero to a working project

```sh
# 1. Create a local project.
fbt init knowledge_ops --template support

# 2. Parse resources and inspect what will run.
fbt parse --project-dir knowledge_ops
fbt doctor --project-dir knowledge_ops
fbt plan --project-dir knowledge_ops --select tag:support

# 3. Build, review, rebuild downstream artifacts, and generate docs.
fbt build --project-dir knowledge_ops --select case_summaries
fbt review approve case_summaries --project-dir knowledge_ops --comment "Reviewed locally"
fbt build --project-dir knowledge_ops --select weekly_support_insights
fbt docs generate --project-dir knowledge_ops
```

This creates `target/artifacts/support/case_summaries/index.md`,
`target/artifacts/support/weekly_insights.md`, local state under `.fbt/state`,
and generated project docs under `target/docs/index.md`.

## Architecture at a glance

```text
project/
├── fs_project.yml        Project contract and paths
├── sources/              Read-only input artifact declarations
├── transforms/           Outputs, refs, runner, assets, policy, evals
├── prompts/              Prompt assets used by LLM/agent runners
├── assets/               Style guides, schemas, rubrics, examples
├── policies/             Read/write/network/review constraints
├── evals/                Deterministic checks and review gates
├── target/               Materialized artifact files
└── .fbt/state/           Manifest, run results, versions, approvals
```
