---
title: External runners
description: How fbt delegates execution outside core.
---

fbt core is not an LLM provider, document converter, or agent runtime. It
delegates execution to external runners.

The user-facing model is one sentence: a runner is an external command that
speaks the fbt runner protocol. That command can wrap OpenAI, Claude Code,
Codex, Gemini, a shell script, a converter, or an internal service.

## Runner examples

- deterministic demo runners for local smoke tests
- command runner for shell-backed transforms
- copyable runner-command scaffold in `examples/runner_adapter_scaffold`
- official OpenAI Responses runner under `adapters/openai`
- adapter packages such as `fbt-runner-claude-code`, `fbt-runner-codex-cli`,
  `fbt-runner-anthropic`, or `fbt-runner-gemini`

## Why the boundary matters

The runner boundary keeps fbt lightweight:

- no provider SDKs in core
- no embedded agent runtime
- no required cloud account
- runner conformance can be tested independently
- adapters can evolve without changing the project graph

## Project configuration

A transform declares the runner it needs. fbt resolves runner commands from
project config, plugin manifests, and `PATH`, then validates capabilities during
doctor/build.

Minimal project-level runner wiring:

```yaml
runners:
  - name: openai.responses
    type: llm
    protocol: stdio_jsonrpc
    command: fbt-runner-openai
    args: ["responses"]
    env:
      - OPENAI_API_KEY
    config:
      provider: openai
      default_model: gpt-5
```

A transform then refers to the logical runner name:

```yaml
transforms:
  - name: incident_response_runbook
    type: llm
    runner: openai.responses
    inputs:
      - source: incident.event_logs
      - source: incident.response_logs
    outputs:
      - name: incident_response_runbook
        type: markdown
        path: target/artifacts/runbooks/incident_response_runbook.md
```

Before a build, `doctor` verifies the command and required environment names.
With a missing key, the failure is explicit and happens before any runner call:

```text
Doctor: error
Project: examples/incident_response_runbook

Runners
  openai.responses
    error RUNNER_ENV_MISSING: runner env OPENAI_API_KEY is not set
    ok RUNNER_COMMAND_OK: runner command is executable: examples/incident_response_runbook/bin/fbt-runner-openai
```

With credentials set, the same project can be planned and built:

```sh
export OPENAI_API_KEY=...
fbt doctor --project-dir examples/incident_response_runbook
fbt plan --project-dir examples/incident_response_runbook --select incident_response_runbook
fbt build --project-dir examples/incident_response_runbook --select incident_response_runbook
```

After build, fbt commits only the declared output candidates and records the
runner fingerprint in the artifact receipt.

## Conformance first

Runner authors and package maintainers should prove protocol compatibility
outside any real project:

```sh
python3 tests/runner-conformance/run.py \
  --runner-command examples/runner_adapter_scaffold/bin/fbt-runner-example \
  --strict \
  --agent-adapter
```

Expected result:

```text
runner-conformance: ok
```

## Official adapter checks

For the adapters maintained in this repository, use three levels of checks:

| Check | Calls a real provider? | Command |
|---|---:|---|
| Fixture/fake protocol smoke | No | `make official-adapter-smoke` |
| Clean install from VCS module paths | No | `make adapter-install-smoke` |
| Installed command plus real credentials | Optional | `make runner-adapter-smoke` |

The source-checkout smoke uses fake responses or fixture CLIs for
`fbt-runner-command`, `fbt-runner-openai`, `fbt-runner-codex-cli`, and
`fbt-runner-claude-code`:

```sh
make official-adapter-smoke
```

Expected result:

```text
official-adapter-smoke: ok
```

Use clean install smoke when you need to prove that nested adapter modules can
be installed without the repository's `go.work` file or local replace rules:

```sh
make adapter-install-smoke
```

That target requires a clean committed working tree because it installs from
the current commit through a temporary bare VCS remote.

## Opt-in smoke matrix

Real provider and CLI-agent checks stay outside `make verify`. To validate
installed runner commands, pass matrix rows to `make runner-adapter-smoke`:

```sh
FBT_RUNNER_ADAPTER_SMOKE_MATRIX='openai.responses|llm|markdown|fbt-runner-openai responses|OPENAI_API_KEY|false
codex.cli|agent|markdown|fbt-runner-codex-cli --profile fbt|OPENAI_API_KEY|true' \
make runner-adapter-smoke
```

Each row runs protocol conformance, a generated-project `doctor`, and a
generated-project `plan`. Set `FBT_RUNNER_ADAPTER_SMOKE_BUILD=1` only when you
intend to call the real provider or agent.
