---
title: External runners
description: How fbt delegates execution outside core.
---

fbt core is not an LLM provider, document converter, or agent runtime. It
delegates execution to external runners.

The user-facing model is one sentence: a runner is an external command that
speaks the fbt runner protocol. That command can wrap OpenAI, Claude Code,
Codex, Gemini, a shell script, a converter, or an internal service.

## Runner examples

- deterministic demo runners for local smoke tests
- command runner for shell-backed transforms
- copyable runner-command scaffold in `examples/runner_adapter_scaffold`
- optional OpenAI Responses runner under `examples/runner_adapters/openai`
- adapter packages such as `fbt-runner-claude-code`, `fbt-runner-codex-cli`,
  `fbt-runner-anthropic`, or `fbt-runner-gemini`

## Why the boundary matters

The runner boundary keeps fbt lightweight:

- no provider SDKs in core
- no embedded agent runtime
- no required cloud account
- runner conformance can be tested independently
- adapters can evolve without changing the project graph

## Project configuration

A transform declares the runner it needs. fbt resolves runner commands from
project config, plugin manifests, and `PATH`, then validates capabilities during
doctor/build.

## Opt-in smoke matrix

Real provider and CLI-agent checks stay outside `make verify`. To validate
installed runner commands, pass matrix rows to `make runner-adapter-smoke`:

```sh
FBT_RUNNER_ADAPTER_SMOKE_MATRIX='openai.responses|llm|markdown|fbt-runner-openai responses|OPENAI_API_KEY|false
codex.cli|agent|markdown|fbt-runner-codex-cli --profile fbt|OPENAI_API_KEY|true' \
make runner-adapter-smoke
```

Each row runs protocol conformance, a generated-project `doctor`, and a
generated-project `plan`. Set `FBT_RUNNER_ADAPTER_SMOKE_BUILD=1` only when you
intend to call the real provider or agent.
