title: Manual generation
description: Practical flow for turning logs into operational procedures.
---

One intended fbt use case is generating formal procedures and manuals from
existing operational evidence:

- incident logs and response logs to an incident response runbook
- customer inquiries and support replies to a support resolution manual
- investigation notes and remediation notes to repeatable procedures

In these examples, fbt owns the build graph, artifact versions, evals, and
lineage. The model call is performed by the configured external runner. Human
approval and publishing happen outside fbt in Git, PRs, CI, release tooling, or
your knowledge-base workflow.

If this is your first custom project, start with the support template and
replace `data/support/tickets/*.jsonl` plus `assets/support_style_guide.md`.
That path proves the local loop with `demo.llm` before you wire a real runner.
The repository guide is `docs/examples/first-own-files-success-path.md`.

For daily operation, keep fbt source paths stable and let cron, CI, Airflow,
Dagster, or another upstream step prepare the current files. The
`examples/daily_qa_ops` project and `docs/examples/daily-source-operations.md`
show multiple source directories producing multiple artifacts in one daily run.

## Incident response runbook

The source checkout includes a practical example that turns one incident's
evidence into an operational runbook:

| Role | File | What the runner must use |
|---|---|---|
| Source | `data/incidents/events/INC-2026-0421.jsonl` | Pager, metrics, mitigation, and resolution events. |
| Source | `data/incidents/response_logs/INC-2026-0421-response.md` | What responders actually did and what gaps remained. |
| Source | `data/incidents/postmortems/INC-2026-0421-postmortem.md` | Root cause and corrective actions. |
| Instruction | `assets/incident_runbook_prompt.md` | Do not invent owners, tooling, thresholds, or system behavior. |
| Format | `assets/incident_runbook_format.md` | Required runbook sections. |
| Runner | `openai.responses` | External OpenAI runner called over the runner protocol. |
| Artifact | `target/artifacts/runbooks/incident_response_runbook.md` | The generated runbook. |

Representative source evidence:

```json
{"incident_id":"INC-2026-0421","source":"pager","event":"p95_latency_ms above 2400 for 10 minutes","customer_impact":"elevated checkout timeouts for a subset of customers"}
{"incident_id":"INC-2026-0421","source":"incident_command","event":"traffic shifted away from us-east-1 read replica","result":"checkout timeout rate decreased from 7.8% to 1.1%"}
```

That evidence is paired with a response note saying support should verify
payment status before asking customers to retry, and a postmortem saying the
root cause was extra read load from a maintenance task. The generated runbook
should therefore contain procedure sections such as:

```md
## Detection
- Treat sustained checkout latency and timeout-rate alerts as incident signals.
- Confirm database connection pool saturation before mitigation.

## Mitigation
- Shift read traffic away from the affected read replica when evidence matches
  the documented saturation pattern.

## Customer Communication
- Acknowledge elevated checkout latency.
- Ask support to verify payment status before recommending a retry.

## Source Evidence
- INC-2026-0421 event log
- INC-2026-0421 response log
- INC-2026-0421 postmortem
```

Preview the work before spending runner time:

```sh
fbt doctor --project-dir examples/incident_response_runbook
fbt plan --project-dir examples/incident_response_runbook --select incident_response_runbook
```

With credentials, the same project calls the optional OpenAI runner:

```sh
export OPENAI_API_KEY=...
fbt build --project-dir examples/incident_response_runbook --select incident_response_runbook
fbt artifact explain incident_response_runbook --project-dir examples/incident_response_runbook
```

The artifact path is:

```text
examples/incident_response_runbook/target/artifacts/runbooks/incident_response_runbook.md
```

`artifact explain` shows the exact source, asset, policy, eval, and runner
fingerprints that will produce or already produced the runbook:


The deterministic eval checks that the required sections exist before the
artifact is committed.

## Support resolution manual

The support example turns accumulated tickets and answer logs into a reusable
support procedure:

| Role | File | What the runner must use |
|---|---|---|
| Source | `data/support/tickets/2026-05-12-login-and-billing.jsonl` | Customer issues, topics, impact, and status. |
| Source | `data/support/response_logs/login-domain-change.md` | Agent handling steps and approved phrasing from real responses. |
| Source | `data/reference/product_docs/account-access.md` | Product rules that constrain the procedure. |
| Instruction | `assets/support_resolution_prompt.md` | Do not invent product capabilities, refund rules, SLAs, or owners. |
| Format | `assets/support_resolution_manual_format.md` | Required support manual sections. |
| Runner | `openai.responses` | External OpenAI runner called over the runner protocol. |
| Artifact | `target/artifacts/support/support_resolution_manual.md` | The generated support manual. |

Representative source evidence:

```json
{"ticket_id":"SUP-10422","topic":"login","summary":"Customer cannot receive password reset email after changing company domain","customer_impact":"blocked from accessing admin console","resolution_status":"resolved"}
{"ticket_id":"SUP-10437","topic":"billing","summary":"Customer asks why seat count increased after SSO group sync","customer_impact":"unexpected invoice estimate","resolution_status":"resolved"}
```

The login response note includes this approved customer-facing explanation:

```text
Your workspace admin needs to update the email on your user profile before the
reset link can be delivered to the new address.
```

The generated manual should turn those scattered records into sections such as:

```md
## Intake Checklist
- Confirm the affected workspace, user email, identity provider status, and
  whether the admin can edit the user profile.

## Resolution Procedure
- For domain-change login issues, ask the workspace admin to update the user
  profile email before resending reset links.

## Customer Response Templates
- Explain that reset links are sent to the email currently stored on the user
  profile.

## Source Evidence
- SUP-10422
- login-domain-change response log
- account-access product documentation
```

## What Makes These Usable

The examples include realistic source records, format assets, scoped policies,
deterministic section evals, and external-runner wiring. The
provider call is outside the default smoke tests because it requires a secret,
but the workflow is shaped for real operational use.

The most important files to inspect are:

| File | Purpose |
|---|---|
| `sources/*.yml` | Declares the evidence files fbt may read. |
| `transforms/*.yml` | Declares the output, model, runner, assets, and evals. |
| `assets/*_format.md` | Defines the required manual or runbook structure. |
| `policies/*.yml` | Limits reads, writes, network, tools, cost, and timeout. |
| `evals/*.yml` | Checks required sections before commit. |
| `target/artifacts/...` | Holds the generated official artifact. |
