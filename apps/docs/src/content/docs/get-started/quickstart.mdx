This page is a control-plane demo. It is not a support-product demo, a model
quality benchmark, or the main manual-generation use case.

The support template is only a small fixture. It gives fbt one source file and
two transforms so you can see the core lifecycle:

| Stage | What you are checking |
|---|---|
| Project checks | fbt can parse the project, resolve runners, and explain the plan. |
| First build | A runner output becomes a versioned artifact on disk. |
| Downstream build | A later transform can consume the current upstream artifact. |
| Inspection/export | fbt can explain local lineage and export OpenLineage/OTel records. |

Use this quickstart to verify that fbt is installed and that the local
filesystem workflow works end to end. After that, use
[manual generation](/fbt/get-started/manual-generation/) for realistic
incident-runbook or support-manual projects, and replace the demo runners with
external OpenAI, Claude Code, Codex, Gemini, or other protocol-compatible
runners.

The command output below was captured from a local `fbt` binary on
2026-05-28. Long paths and hashes are shortened.

## 1. Create and Check a Project

This step proves that the project definition is valid and that fbt can tell you
what will run before any artifact is generated.

```sh
fbt init knowledge_ops --template support
fbt doctor --project-dir knowledge_ops
fbt plan --project-dir knowledge_ops --select tag:support
```

Expected output:

```text
Initialized support project at <tmp>/knowledge_ops
Files: 12
Demo runners: configured as demo.*; replace runner commands in fs_project.yml for real provider execution

Doctor: ok
ok PROJECT_CONFIG_OK: project config parsed
ok STATE_LOCK_OK: state lock can be acquired and released
ok RUNNER_PROTOCOL_OK: runner protocol initialize succeeded
ok RUNNER_CAPABILITIES_OK: runner capabilities satisfy configured transforms

Plan: 2 selected, 1 run, 0 skipped, 1 blocked
run transform.knowledge_ops.case_summaries
  reason: no previous successful run
  reason: output missing
blocked transform.knowledge_ops.weekly_support_insights
  blocked: requires artifact.knowledge_ops.case_summaries current artifact
  next: fbt build --select case_summaries
```

The plan is telling you that the first transform can run immediately, but the
downstream weekly insight is blocked until `case_summaries` exists.

## 2. Build the First Artifact

This step proves that fbt can call a configured runner and commit the runner's
candidate output as an artifact version.

```sh
fbt build --project-dir knowledge_ops --select case_summaries
```

Expected output:

```text
Build: 1 selected, 1 run, 0 skipped, 0 blocked
success transform.knowledge_ops.case_summaries
  committed: artifact_version.knowledge_ops.case_summaries.sha256_a5b4...
```

The current artifact is written to:

```text
knowledge_ops/target/artifacts/support/case_summaries/index.md
```

fbt also stores an immutable copy under `.fbt/artifacts` and records the run in
`.fbt/state`.

Generated artifact excerpt:

```md
# Case Summaries

Generated by the deterministic FBT demo LLM runner.

- transform: case_summaries
- provider: demo
- model: deterministic-demo-llm
- mode: demo fixture, not a provider call
```

## 3. Build Downstream

This step proves that fbt can pass a current upstream artifact into a later
transform. fbt does not implement human approval; use Git, PRs, CI, release
tooling, or your knowledge-base workflow when people need to approve the
generated files.

```sh
fbt build --project-dir knowledge_ops --select weekly_support_insights
```

Expected output:

```text
Build: 1 selected, 1 run, 0 skipped, 0 blocked
success transform.knowledge_ops.weekly_support_insights
  committed: artifact_version.knowledge_ops.weekly_support_insights.sha256_49f...
```

The second output is:

```text
knowledge_ops/target/artifacts/support/weekly_insights.md
```

Generated downstream excerpt:

```md
# Agent Run: Weekly Support Insights

## Plan

- inspect inputs
- apply declared tools
- write candidate artifact
```

## 4. Inspect Lineage Locally

This step proves that the generated file is not just a loose Markdown file.
You can ask fbt where it is stored, which transform produced it, what version
is current, and what the downstream plan now sees.

```sh
fbt artifact path case_summaries --project-dir knowledge_ops
fbt artifact history case_summaries --project-dir knowledge_ops
fbt artifact explain weekly_support_insights --project-dir knowledge_ops
```

Expected output:

```text
logical_path: target/artifacts/support/case_summaries
storage_path: .fbt/artifacts/artifact_version.knowledge_ops.case_summaries.sha256_a5b4.../content
version: artifact_version.knowledge_ops.case_summaries.sha256_a5b4...

artifact.knowledge_ops.weekly_support_insights
  transform: transform.knowledge_ops.weekly_support_insights
  action: skip
  input: ref artifact.knowledge_ops.case_summaries
  current_version: artifact_version.knowledge_ops.weekly_support_insights.sha256_49f...
  confidence: structural
  previous_run: transform_run.run_... success
  next: fbt artifact show weekly_support_insights
```

## 5. Export Standard Records

This step proves that fbt does not require a custom visualization backend.
It exports local run and artifact state into standard-compatible files.

```sh
mkdir -p knowledge_ops/target/lineage knowledge_ops/target/telemetry
fbt export openlineage \
  --project-dir knowledge_ops \
  --output knowledge_ops/target/lineage/openlineage.ndjson
fbt export otel \
  --project-dir knowledge_ops \
  --output knowledge_ops/target/telemetry/otel.json
```

Expected output:

```text
OpenLineage events written to knowledge_ops/target/lineage/openlineage.ndjson
Events: 2

OTel traces written to knowledge_ops/target/telemetry/otel.json
Spans: 4
```

The final files created by the quickstart are:

```text
knowledge_ops/target/artifacts/support/case_summaries/index.md
knowledge_ops/target/artifacts/support/weekly_insights.md
knowledge_ops/target/lineage/openlineage.ndjson
knowledge_ops/target/telemetry/otel.json
```

## What to Replace for Real Work

The quickstart runners are named `demo.llm` and `demo.agent`. Their output is
intentionally small because this page is testing fbt's control plane, not model
quality.

For real work:

| Replace | Keep |
|---|---|
| `demo.llm` or `demo.agent` runner commands | `doctor`, `plan`, `build`, `artifact`, and `export` workflow |
| Fixture support ticket data | Source declarations, artifact paths, policies, and evals |
| Demo Markdown output quality | Artifact versions, local lineage, and standard exports |

For operational manuals and runbooks, start from
[manual generation](/fbt/get-started/manual-generation/) rather than this
fixture.
